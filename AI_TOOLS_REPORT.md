## 3. Phase-by-Phase Usage

### 3.1 Planning & Backlog (Sprint 0)

AI tools were used to accelerate the initial product backlog and help structure user stories.

**How:** The team used ChatGPT and Claude Web to iterate on user story definitions and acceptance criteria. The team provided the product vision and constraints; AI helped formalize them into 15 user stories across 6 epics (User Management, CV Parsing, Job Preferences, Job Finder Agent, CV Tailor Agent, Digest & Tracking).

A `project-context.md` file was created to serve as persistent context for AI coding assistants, containing the tech stack, backlog, and architectural decisions — a technique that improved the coherence of AI-generated code across multiple sessions.

**Outcome:** A complete, well-structured backlog generated collaboratively in a fraction of the time it would have taken manually.

---

### 3.2 Architecture & Design

AI was consulted to validate architectural decisions and review design tradeoffs.

**How:**
- The three-tier architecture (React frontend, Django REST API, PostgreSQL) was designed by the team and validated with Claude, which flagged potential CORS misconfiguration risks early.
- Component architecture, context, and workflow diagrams (`/diagrame/`) were created using AI-assisted diagramming tools based on natural-language descriptions.
- The two-level caching strategy (per-source DB cache with TTL + browser sessionStorage) was a team decision. The specific implementation detail — using an MD5 hash of `(job_title | location | source)` as the cache key — emerged from an AI-assisted discussion of the tradeoffs.

### 3.3 Implementation

This is where AI tools had the highest impact. A significant portion of the codebase was written with direct AI assistance via Claude Code, while the team retained ownership of product decisions and code review.

#### Backend (Django)

**AI-assisted components include:**

- **`JobScoringAgent`** (`backend/app/jobs/ai_agent.py`) — The agent class, including the prompt structure for structured JSON output and the fallback regex parser for when the LLM returns slightly malformed JSON, was written with Claude Code guidance.
- **`CVTailorAgent`** (`backend/app/cv/cv_tailor_agent.py`) — The diff-based change set logic (`_normalize_section_changes`, `_resolve_section_items`) was largely AI-generated. The constraint that the agent must never fabricate skills was an explicit product decision encoded into the prompt.
- **`CoverLetterAgent`** (`backend/app/cv/cover_letter_agent.py`) — Prompt structure, output format, and the markdown code fence stripping logic were implemented with AI assistance.
- **Job caching system** (`backend/app/jobs/views.py`) — The per-source cache lookup loop and TTL mechanism were generated by Claude Code after the team described the desired caching behavior.
- **`wait-for-db.sh`** and Docker health-check configuration were generated by AI and reviewed by the team.

#### Frontend (React)

- The component structure and CSS design system were developed iteratively — the team made design decisions, Claude Code implemented them.
- The `jobsSearchCache.js` two-level cache (sessionStorage + in-memory) was AI-generated.
- The `TailoredCVPage` diff review UI was implemented with AI assistance.
- The UI redesign (professional color palette, Inter font, SVG icons, ambient gradient background) was done collaboratively: the team provided visual direction, Claude Code applied the changes.

#### Prompt Engineering for In-Product Agents

Significant effort went into crafting prompts that produce reliable, structured outputs from a small local model (`qwen2.5-coder-1.5b-instruct`, 1.5B parameters). Key techniques:

- **Strict output format instructions** — prompts explicitly require valid JSON with no markdown wrapping.
- **Hard constraints in the prompt** — e.g., *"NEVER invent skills, experience, or qualifications not in the original CV"* prevents hallucination in the CV tailor.
- **Temperature tuning** — `0.1` for scoring (deterministic), `0.3` for tailoring, `0.5` for cover letters.
- **Fallback regex parsing** — because small models occasionally produce slightly malformed JSON, a regex fallback was added to extract the score and summary even when `json.loads` fails.

The prompt engineering was a collaborative process: the team defined the behavioral constraints (no hallucination, specific output schema), and AI helped translate them into effective prompt language.

### 3.4 Testing

AI tools assisted in scaffolding the test suite, while the team defined what to test.

**How:**

*Backend (pytest-django, pytest-mock) — 29 tests:*
- `users/tests/test_auth.py` — 5 tests covering user model creation, login endpoint, and `/me` profile endpoint.
- `cv/tests/test_views.py` — 2 tests for the tailored CV download endpoint (happy path + 404).
- `jobs/tests/test_search.py` — 2 tests for the job search endpoint, mocking both external fetchers and the `JobScoringAgent`.
- `jobs/tests/test_agent_evals.py` — 5 eval tests for `JobScoringAgent`: valid JSON response, markdown-fenced JSON, malformed JSON with regex fallback, network error returning score 0, and score range validation.
- `cv/tests/test_cv_tailor_agent.py` — 10 eval tests for `CVTailorAgent`: unit tests for `_normalize_section_changes`, `_resolve_section_items`, and `_build_result`, plus integration evals for the `tailor()` method with mocked HTTP responses covering happy path, markdown fence stripping, network error fallback, and the no-fabrication constraint.
- `cv/tests/test_cover_letter_agent.py` — 5 eval tests for `CoverLetterAgent`: happy path, markdown fence stripping, network error fallback, missing candidate name, and no-markdown-formatting in output.
- `conftest.py` — shared fixtures (`user`, `auth_client`, global HTTP blocker) were generated by AI and reused across all test modules.

*Frontend (Vitest + Testing Library) — 12 tests:*
- `Navbar.test.jsx` — renders brand and navigation links.
- `JobsPage.test.jsx` — renders job results after clicking Search.
- `AuthPage.test.jsx` — 4 tests: login form shown by default, Register tab shows Full name field, server error displayed, network error displayed.
- `ApplicationsPage.test.jsx` — 3 tests: empty state, jobs grouped into Applied/Saved/Rejected columns, API error displayed.
- `PreferencesPage.test.jsx` — 3 tests: preferences loaded from API, success message on save, last active job source cannot be disabled.

**Eval for AI agents:** The three eval test files (`test_agent_evals.py`, `test_cv_tailor_agent.py`, `test_cover_letter_agent.py`) are proper agent evals — they mock `requests.post` with realistic LLM responses (valid JSON, malformed JSON, network errors) and assert that each agent correctly parses, processes, and falls back. This is distinct from simply mocking the agent class: the agent logic itself is exercised in every eval test.

### 3.5 CI/CD Pipeline

The GitHub Actions pipeline (`.github/workflows/main.yaml`) was designed by the team and implemented with extensive AI assistance.

**Pipeline structure:**
1. **Backend CI** — flake8 lint + pytest with a real PostgreSQL service container
2. **Frontend CI** — ESLint + Vitest (runs in parallel with backend CI)
3. **Docker sanity build** — runs only after both CI jobs pass
4. **CD deploy** — builds and pushes Docker images to GHCR on merge to `main`

One notable debugging instance: the initial pipeline failed on GHCR because the repository name `P4ulCun/FindMyJob` contains uppercase letters. Claude Code identified the issue and suggested the `tr '[:upper:]' '[:lower:]'` normalization step, which fixed the deployment job.

### 3.6 Debugging & Code Review

Claude Code was used throughout the project for debugging:

- **LM Studio connectivity** — diagnosed a model name mismatch (`qwen2.5-1.5b-instruct` vs the loaded `qwen2.5-coder-1.5b-instruct`) causing 60-second timeouts in cover letter generation. The AI analyzed Docker container logs and ran connectivity tests from inside the container to identify the root cause.
- **Merge conflict** — an unresolved git conflict left in `CoverLettersPage.css` (with `<<<<<<<` markers in the file, causing broken CSS) was detected and resolved.
- **Docker environment variable issue** — when a new env var wasn't picked up after `docker compose restart`, the AI correctly explained that `restart` does not recreate containers and suggested `--force-recreate`.

## 5. Key Observations

**What worked well:**
- The `project-context.md` file eliminated the need to re-explain the architecture at the start of every AI session, significantly reducing repeated context setup.
- AI was most effective at boilerplate-heavy code (serializers, URL routing, Docker configuration) and at translating high-level descriptions into working first drafts.
- Using AI for prompt engineering of the in-product agents was particularly valuable — iterating on prompt structure with AI feedback was faster than pure trial-and-error against a live LLM.
- AI caught several bugs early (merge conflict in CSS, model name mismatch, container restart behavior) that would have taken significant time to debug manually.

**Where human judgment was essential:**
- Deciding on cache invalidation policy (product decision, not a coding problem).
- Evaluating the quality of AI-generated cover letters from the small local model and tuning prompts based on output quality.
- Resolving merge conflicts where both branches contained valid changes.
- Defining UI direction — AI could implement changes but required clear human direction on what "professional" meant.
- All code review: AI-generated code was reviewed before merging to ensure it matched the intended behavior and had no security issues.

## 6. Conclusion

AI tools, primarily Claude Code, were a meaningful part of the development workflow. The team adopted a pattern of AI-assisted implementation — describing features in natural language, reviewing AI-generated drafts, and refining collaboratively. This approach allowed a 4-person team to deliver a full-stack application with 3 AI agents, a CI/CD pipeline, a test suite, and a polished UI within a single semester.

The most significant productivity gains came from:
1. **Context persistence** via `project-context.md` keeping the AI coherent across sessions
2. **Infrastructure generation** — Docker, CI/CD, and test boilerplate written correctly on the first attempt
3. **Debugging assistance** — AI could analyze logs and propose root causes faster than manual investigation

The team's role shifted toward product ownership, design direction, code review, and integration — tasks where domain knowledge and judgment matter more than typing speed.
